### The decisions that shaped it

Most of the choices that made Pawlog what it is were about deciding what not to build. The obvious path for versioning was a custom audit table with version IDs and rollback logic. But git had already solved that problem, and solved it better than I would have. So the copy folder is its own git repository, and every save shells out to `git add` and `git commit`. The tool just puts a friendly face on something git already does. You can run `git log` in that folder without touching the app at all.

The copy repository is also kept separate from the app code, which matters more than it sounds. If app changes and copy changes share a single history, neither is readable. A writer reviewing what changed to the homepage headline shouldn't have to scroll past commits about CSS. The two repos are entirely independent, which means the copy history stays clean and can be pushed to a remote for backup on its own schedule.

The stack follows the same logic. Alpine.js off a CDN, plain CSS with custom properties, no build step. A local copy-editing tool doesn't need a reactive component tree, and it took a small amount of judgment to not add one anyway. The constraint keeps the whole thing maintainable by one person.

Each page of copy is a single JSON file with a shallow, predictable structure and snake_case keys that developers can reference safely in code. When you run a diff, you can read exactly what changed. No migration scripts, no schema versions to manage.

The terminology was also a deliberate choice. The previous model of channels, sections, and widgets didn't match the way writers think about their own work. Journey, Page, Section maps more naturally to how a product is actually structured, and the words a tool uses quietly shape how people think about the thing they're managing.

Saves only ever move forward. No amend, no force-push, no overwrite. If someone saves the wrong thing, the previous version is always one commit back. History is the safety net and it can't be removed by accident.

The export formats came last and were the most user-facing decision. The process Pawlog was replacing ran on Word documents with tracked changes, passed around in Teams chats. Asking compliance reviewers to learn a new workflow was the fastest way to get the tool ignored, so pages and journeys export as `.docx` files with an optional section filter and optional revision history pulled from the git log. JSON export and import sit alongside it for the developer handoff case. Two formats for two audiences, from the same underlying data.

> Version history isn't a feature you build. It's a primitive you expose. Most copy tools either ignore history or write their own inferior version of it. The bet of Pawlog was that git had already solved that problem better than I could — and the same history a developer reads as a diff is the revision log a reviewer reads at the end of a Word document.

### What it does

The dashboard is the first thing you see, with live counts of journeys, pages, and copy items, plus a feed of recent commits so you can tell at a glance what's been touched. The sidebar gives you the hierarchy — journeys collapsing open to reveal their pages — with full create, rename, and delete for both. Selecting a page brings up every copy item grouped by section, and selecting an item slides a panel over the top so you can edit the string, leave a commit note, and save without losing your place.

The history tab shows the git log for that page, and you can pick two commits and read a colour-coded diff inline, so a writer can see exactly what changed and when without leaving the browser. Export works in two shapes for two audiences: JSON across a single page, a whole journey, or all of the copy at once, with every import landing in history like any other save; and Word for compliance review or sign-off, with the embedded revision history meeting reviewers in the format they already use.

### Where I'd push back on myself

This started as a personal project to scratch a personal itch, and that is also its weakness. I went from "this annoys me" to building working prototypes fairly quickly, when I could have done more research into how a tool like this might help other teams before I started making technical decisions. I could have gathered most of the feedback I needed with just a working UI, or even a Figma prototype, rather than a running backend.

That said, the backend turned out to be so quick to build — a few hours, really — that I don't regret it entirely. I demoed the tool periodically as it came together and collected feedback and feature requests from colleagues along the way, so it didn't develop in a vacuum even if it didn't begin from formal research.

It's currently being trialled internally and the reviews have been positive. There's genuine room for a system like this, especially if product and engineering can agree on a standard set of copy keys. If they do, a copy change becomes as simple as exporting and importing a JSON file, which was the whole point.
